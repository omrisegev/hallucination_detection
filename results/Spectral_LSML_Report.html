Unsupervised Spectral Hallucination Detection

A single-forward-pass detector built on the spectral structure of the token-entropy trace H(n).  ·  Analysis over 29 cached model×dataset evaluation cells · June 2026

This report reviews what the past phase of work established: which design choice was actually responsible for the performance we lost when the method was made fully unsupervised, how the algorithm behaves internally, and how it stands against the literature. It is organised as a sequence of questions; every table and chart is here to answer one of them.

Read this first0. Terminology and how the numbers are aggregated

The analysis varies the method along three independent axes. Every variant named below is one choice on each axis:

  • Feature set — how many H(n) features feed the fusion: 5 (the curated subset GOOD_5 = epr, low_band_power, sw_var_peak, cusum_max, spectral_entropy), 9 (STABLE_H9 — the H(n) features that rarely saturate), or 16 (all H(n) features, no selection).
  • Encoding — binary (each feature median-split to ±1, as the source papers assume) vs continuous (each feature z-scored, magnitude preserved).
  • Fusion — flat SML (Spectral Meta-Learner; assumes inputs are independent), L-SML (Latent SML; first clusters the dependent features, fuses within each cluster into a "virtual classifier," then across clusters), or a plain simple average baseline.

How results are aggregated — three levels, used consistently:

  • A cell is one model × dataset evaluation (e.g. MATH-500 on Qwen2.5-Math-7B). There are 29 cells.
  • A domain mean averages all cells in a domain (MATH-500 has 4 cells, RAG has 16, etc.).
  • Macro is the mean of the five domain means — each domain weighted 1/5 regardless of cell count.
Which level each section uses: the analysis sections (Findings 1–4, feature stability, the per-domain ranking) report domain means / macro. The cluster examples (§7) are per individual cell, because clusters are formed per cell. The benchmarking tables (§10) are per cell, each matched to the exact model the published/in-house baseline ran on — that is what a model-matched comparison requires.

What is being measured, and what went wrong1. The detector, and the three changes that caused the drop

For each generated answer we record the per-token entropy sequence H(n) of the model's output distribution — one number per token, from a single forward pass (we never re-run the model; this is also the answer to the runtime question — we store the entropy trace, not the full logits). From H(n) we compute 16 spectral/temporal features (band powers, spectral entropy, entropy-production rate, sliding-window variance, CUSUM statistics, etc.) and fuse them into one correctness score. No labels are used at inference.

When the pipeline was made faithful to the papers, three things changed at once:

  1. Binary vs continuous. The SML theory assumes each input is a binary ±1 classifier; the earlier implementation fed it continuous (normalized) features.
  2. SML vs L-SML. Plain SML assumes the inputs are independent. Our 16 features all come from the same entropy trace, so they are dependent — L-SML first clusters the dependent features, fuses within each cluster, then across clusters.
  3. Label leakage. The old pipeline used ground-truth labels in two places: to search over feature subsets for the best-AUROC fusion, and to set each feature's sign/direction. Both were removed.
The question this report answers: of these three changes, which one actually caused the performance to fall — and how much of it can be recovered honestly? To find out, we ran ~18 variants of the method over all 29 stored cells, changing one axis at a time. No new inference was needed — everything was recomputed from the stored entropy traces.

Takeaway: a single change — continuous instead of binarized features — recovers +4.9pp macro and up to +12pp on individual domains. Median-binarization collapses each feature to "above/below median," discarding the magnitude information that actually separates correct from hallucinated generations. This, not the removal of label leakage, was the main cause of the drop. (Finding 4 shows continuous beats binary in every cell of the full grid, so this is not specific to the 5-feature set.)

Finding 2 — answering "is the subset necessary?"3. Feature selection is a minor convenience, not a requirement

One of the two leakage sources was searching, with labels, for the best feature subset. A fair question (raised directly by the advisors): does the method depend on that hand-picked subset at all? We compare the curated 5-feature set against running continuous L-SML on the 9-feature stable set and on all 16 features — same encoding, same fusion, only the feature count changes (all macro AUROC):

Takeaway: continuous L-SML lands at 68–70% whether it uses 5, 9, or all 16 features — dropping feature selection entirely costs at most ~2pp (and only 0.9pp at 16). The method does not depend on a label-selected subset; the 5-feature set is a small, consistent optimisation on reasoning traces, not the source of the result. (Why L-SML can absorb the extra 4 or 11 features without harm is Finding 4 — and §7 shows the mechanism directly.)

Finding 3 — the second leakage source4. Feature "directions" collapse to one physical prior

The other leakage source was using labels to orient each feature (does a high value mean "more likely correct" or "more likely wrong"?). Two results show this is not a learned, label-dependent dictionary:

(a) In continuous mode, the offline directions change the AUROC by exactly zero. The variant with our signs and the variant with no signs produce identical AUROC in all 29 cells — because applying the signs (all five point the same way) is mathematically just a global sign-flip of the final score, which AUROC is invariant to. So the directions add no discriminative power; their only job is orientation.

(b) The paper's own unsupervised orientation rule does not work for our features. The L-SML "assumption (iii)" orients by assuming most classifiers beat random at predicting the correct class. Our features are uncertainty measures — they predict the error class — so the rule points the wrong way. Measured directly, the algorithm's internal direction is correct in only:

Variant (no external signs)Cells with correct internal direction
L-SML, 16 features, binary31% (9/29)
L-SML, 5 features, binary21% (6/29)
L-SML, 5 features, continuous14% (4/29)
Takeaway: orientation cannot be recovered from unlabeled covariance alone, but it does not require labels either. It reduces to one physical fact — "higher entropy ⇒ more likely a hallucination" — supplied once, globally. That is domain knowledge, not a per-feature dictionary learned from the test set.

Two things are visible. First, continuous beats binary in every single cell of the grid (Finding 1 is universal, not specific to one feature set). Second, the value of clustering grows with the number of features: in continuous mode L-SML ties flat SML on the clean 5 (70.1 vs 70.0), beats it by +3.6 on 9 features and +6.1 on 16, and beats the simple average by +2.9 on 16. Takeaway: on the clean 5-feature set the fusion choice barely matters — the encoding does the work. But L-SML is the only method that does not fall apart as features are added (it stays 68–70% from 5 to 16 features, while flat SML collapses 70 → 63). That robustness to redundant features is exactly what makes Finding 2 possible: you can skip feature selection because L-SML quarantines the noise.

Why clustering is the right tool6. The features are dependent — the structure flat SML ignores

Flat SML assumes the inputs are conditionally independent. All 16 features are derived from the same entropy trace, so that assumption is false — and the violation is not uniform. The heatmap below is the mean |Spearman ρ| over all 29 cells for every one of the 120 feature pairs (darker = more dependent). It is the structure L-SML is built to exploit and flat SML is built to assume away.

Feature dependence — mean |Spearman ρ| across 29 cells (all 120 pairs)

Median pair correlation is 0.25, but a tight band-power block stands out: hl_ratio · spectral_centroid 0.88, high_band_power · hl_ratio 0.84, low_band_power · hl_ratio 0.77 — 5 pairs exceed 0.75. Meanwhile pe_mean correlates weakly with everything (max 0.55, mostly <0.25).

Takeaway: the dependence is block-structured, not flat — a handful of highly redundant band-power features, plus near-independent outliers like pe_mean. Flat SML double-counts the redundant block and is dragged by the outliers; L-SML groups the block into one virtual classifier and isolates the outliers. That is the mechanism, and §7 shows it operating per cell.

Opening the box7. The clusters L-SML forms, and how it suppresses weak features

L-SML's claim is mechanistic: it groups dependent features, fuses each group into a virtual classifier, then weights the groups when fusing across them. We can check all of this directly — score each cluster against the labels (its AUROC), and read off the across-group weight the algorithm assigns it (normalised to sum to 1 over the clusters in a cell; a near-zero weight means the cluster is effectively switched off). So the band-power block (the dark square in §6) is collapsed into one 94.7% cluster instead of being counted five times, the other informative features form two more ~93% clusters, and the weak outlier pe_mean is isolated and given a weight of 0.02 — the algorithm suppresses it automatically.

Does pe_mean's weight tell us to delete it? Only sometimes — and that is the point

A natural reading is "pe_mean is always isolated at ~55% with weight ~0 — just remove it." That is true on some cells but not all, and the difference is informative. Here is pe_mean across three reasoning cells, 9-feature continuous L-SML:

Takeaway: where pe_mean is weak (MATH, both QA cells) L-SML isolates it and assigns its cluster a weight of 0.02–0.05 — it is already switched off, so deleting it by hand would change essentially nothing. But on GSM8K pe_mean is dependent enough on epr to join a useful 67.7% cluster and earns a real weight (0.24). Its value is domain-dependent, and the across-group weight handles that adaptively per cell. That is a stronger argument than a static "remove pe_mean": you do not need to — L-SML's weighting already does it where it should, and only there.

Where the method works, and why8. It is a reasoning-trace method — and no single feature is both strong and stable

The features are not uniformly useful, and their usefulness is not portable across domains. The scatter plots every feature's mean AUROC (over the five domain means) against its spread across domains. The pattern is stark: the strongest features are the least stable.

Feature strength vs cross-domain stability (each point = one H(n) feature)

Up-and-right would be the ideal (strong and stable); it is empty. The strong features (epr 68.1 mean / 29.5 range, cusum_max 68.3 / 25.6, sw_var_peak 67.3 / 26.0) swing ~30pp between domains — ~84% on math, ~54% on GPQA. The stable features (pe_mean range 8.5, cusum_shift_idx 11.0) are weak everywhere. The five GOOD_5 features are highlighted.

Continuous L-SML (5-feature) by domain

That no feature is both strong and stable is the structural reason fusion helps on reasoning and not on short-answer tasks. The method is strong on reasoning traces (MATH-500, GSM8K, and chain-of-thought factual QA) and weak on GPQA (short forced-choice) and RAG (retrieval-grounded) — those traces are short and lack the token-level temporal evolution the spectral analysis relies on. Restricted to the three reasoning domains, the 5-feature continuous method reaches a 78.3% macro — and there it beats not only a simple average but also the best1 oracle that picks the single best feature per cell with labels, confirming the fusion adds real signal where the method applies.

Honest limitation: on GPQA the continuous 5-feature method is actually our weakest variant (53.9); the 16-feature variant does a little better (56.7) but still trails the baselines. We treat GPQA and RAG as outside the method's operating regime rather than claim them.

What we now know, and what is next11. Conclusions

Questions answered by these experiments:

  1. What caused the post-correction drop? Binarizing the features — not removing the label leakage. Continuous encoding recovers +4.9pp macro / +7.2pp on reasoning (§2), and is better in every cell of the grid (§5).
  2. Does the method need a hand-picked feature subset? No — continuous L-SML is 68–70% at 5, 9, or 16 features (§3). Selection is a ≤2pp convenience.
  3. Are the feature directions a label-dependent trick? No — they collapse to one physical prior ("higher entropy ⇒ hallucination") and add zero discriminative power in continuous mode (§4).
  4. Does L-SML's clustering matter, and how? Yes, with many features — it collapses the redundant band-power block into one virtual classifier and switches off weak outliers via a near-zero across-group weight (§6–§7), which is what lets it run on all 16 features without degrading.
  5. Should we delete pe_mean? No need — L-SML already gives it weight 0.02–0.05 where it is weak, and keeps it only where it helps (§7).
  6. Where does it work? Reasoning traces (78.3% macro on MATH-500 / GSM8K / QA, beating the oracle single feature); not GPQA or RAG, which lack the temporal structure (§8–§9).

Main configuration: continuous L-SML on the 5-feature set, with a single global orientation prior — fully unsupervised, one forward pass. Next steps: fix the grading bug in the EDIS comparison and complete the stronger-model GPQA comparison (Phase 14); then consider conformal calibration of the score for a deployable abstention threshold.
